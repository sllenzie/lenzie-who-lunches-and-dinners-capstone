Capstone reflection

Why I Built This Project

Dining recommendations are often shared casually, but they carry so much detail: the atmosphere, the context, the neighborhood energy, and the small menu decisions or interactions that make a meal more than just the food on the plate. I wanted to preserve that context instead of reducing each place to a rating or star system, and part of how I did this was to only include locations I had a positive enough experience at to remember it, even years later.

Originally, this started as a class lab designed to introduce Mapbox, geocoding, and location-based data. I turned that exercise into a larger restaurant directory by building an initial dataset of more than fifty Los Angeles restaurants, giving the project a real purpose beyond the assignment prompt.

When the capstone phase began, I did not want to abandon the work I had already built. After checking in with my professors, I got permission to expand Lab 06 instead of starting from scratch, which let me focus on improving the data model, user experience, and visual presentation all in one place.

The expansion also became a way to document the places I have been fortunate enough to eat around the world. I used old credit card statements and memory to recover names, then organized the records so the site could present each restaurant consistently through cards, filters, and map markers.

Design reasoning

Clarity first, personality second

I wanted the interface to feel curated without becoming cluttered. The layout uses large imagery, strong hierarchy, and restrained text blocks so the restaurants remain the focus while the design supports the content.

That also shaped the interaction model. Cards surface the essential details, popups keep the map useful, and the modal view lets someone inspect a restaurant without losing their place in the list or on the map.

Because the content lives in CSV files, the site stays easy to maintain. Adding or updating a restaurant only requires editing the dataset instead of rebuilding the interface, which made it practical to keep growing the collection over time.

Technical focus

What I built

The project combines front-end development, data processing, and map visualization in one workflow. The site loads restaurant records from CSV, parses them in the browser, and turns them into cards, filters, and interactive map markers.

On the backend side of the project, I used Python to geocode the addresses and generate a richer CSV with latitude and longitude values. That step made the mapping layer possible and also kept the dataset reusable for future updates.

On the front end, I used HTML, CSS, and vanilla JavaScript with Leaflet and Mapbox tiles to build an interface that responds to different ways of browsing: by cuisine, attire, location, or by clicking directly on the map.

Overall, the site reflects how I like to approach product work: define the user question first, organize the data so it can support that question, and then design the interface so the answer is easy to explore.
